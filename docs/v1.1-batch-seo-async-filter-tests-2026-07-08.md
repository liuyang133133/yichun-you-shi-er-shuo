## 🎯 Part 1: Batch SEO 异步化 (P1)

### 根因

`POST /admin/ai/regenerate-seo-batch` 同步调用 `batchGenerateSeoMeta`, 单条 LLM 调用 ~5s, MAX_BATCH=20 → **最坏 100s**。HTTP event loop 被阻塞 100s, 期间 NestJS 无法处理其他请求, 易触发 nginx/客户端 timeout。

### 修复: 轻量异步 (不引入 BullMQ)

**`seo.service.ts`** — 加 in-memory job Map + setImmediate 异步跑:

```ts
private readonly batchJobs = new Map<string, BatchJob>();

submitBatchJob(limit = 50): { jobId, status: 'pending', limit } {
  const jobId = `seo_batch_${Date.now()}_${random6}`;
  this.batchJobs.set(jobId, { jobId, status: 'pending', limit, ... });
  // 限制 Map 大小 100, 防止内存泄漏
  setImmediate(() => this.runBatchJob(jobId, limit).catch(...));
  return { jobId, status: 'pending', limit };
}

getBatchJobStatus(jobId) { return this.batchJobs.get(jobId) ?? null; }
```

**`seo.controller.ts`** — 改 batch 端点 + 加状态端点:

```ts
// POST: 立即返 jobId
@Post('admin/ai/regenerate-seo-batch')
async batchRegenerate(@Body() body: { limit?: number }) {
  return this.service.submitBatchJob(body.limit ?? 50);  // < 1ms
}

// GET: 查 job 状态
@Get('admin/ai/regenerate-seo-batch/:jobId')
async getBatchJobStatus(@Param('jobId') jobId: string) {
  const job = this.service.getBatchJobStatus(jobId);
  if (!job) throw new NotFoundException(...);
  return job;
}
```

**保留**: 旧的 `batchGenerateSeoMeta()` 方法仍存在 (供 cron `dailyBatch` 用, 同步跑 OK)。

### 权衡

| 方案 | 优点 | 缺点 |
|---|---|---|
| BullMQ 队列 | 持久化 / 跨实例 / 重试 | 新依赖 + 新 module + 改 cron |
| **轻量 in-memory (采用)** | 5 行改动, 不引入新依赖, 立即返 | **单实例 + 重启丢任务** (V1.1 接受) |

升级路径: V1.2 替换为 BullMQ, 接口不变.
